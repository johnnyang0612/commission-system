# 川輝科技內部系統升級需求分析與功能規劃

**作者：Manus AI**  
**日期：2026-04-03**

## 一、文件目的

本文件根據您目前的內部系統架構，以及您新提出的兩大痛點，整理成一份可直接拿去內部討論、交付工程規劃，或後續延伸為 PRD / 提案 / 對外開發規格的升級需求分析文件。核心目標有兩個。第一，是把**合約、建案、付款、分潤、勞報單與簡易帳務**從人工輸入轉向半自動或自動化；第二，是把**川輝 AI 助理**從「會記錄與摘要」升級為「能辨識、能操作、能協作、能回答」的群組智能助理。

| 主軸 | 升級目標 | 對公司價值 |
|---|---|---|
| 合約與帳務自動化 | 上傳文件後自動建案、建付款期程、建驗收點、計算分潤與產勞報單 | 降低人工輸入與財務錯誤，提高案件可追蹤性 |
| 川輝 AI 助理升級 | 提高成員辨識、支援建會議、檔案取回、知識問答、手動補正 | 提升群組協作效率，讓 LINE 成為真正的工作入口 |

## 三、需求主軸 A：合約、建案、付款、分潤、勞報單自動化

### 3.1 目標描述

您希望未來當內部人員上傳**最終提案書 + 合約**後，系統不是只存檔，而是能直接把這份文件變成一個可執行的案件。也就是說，系統必須從文件中抽出商務資訊、專案資訊、付款資訊、驗收資訊、保固與維護資訊，再轉成系統內可追蹤的結構化資料。

### 3.2 建議的自動建案流程

| 階段 | 系統動作 | 使用者介面 |
|---|---|---|
| 上傳文件 | 上傳最終提案書、合約、補充條款 | 單一「AI 建案」入口 |
| 文件解析 | 擷取客戶名稱、專案名稱、總金額、付款條件、驗收點、保固、維護費 | 顯示抽取結果預覽 |
| AI + 規則校正 | 把抽取結果標準化成案件主資料、分期、里程碑 | 顯示信心分數與待確認欄位 |
| 人工確認 | 對 AI 不確定欄位進行快速修正 | 一頁式確認畫面 |
| 正式建案 | 建立 project、installments、milestones、warranty、maintenance plan、commission rules | 建案成功並導向專案詳情 |
| 後續自動化 | 後續收款、分潤、勞報單、維護費依規則自動推進 | 自動計算、可人工覆寫 |

### 3.3 合約抽取的欄位設計

建議把「AI 抽取結果」與「正式案件資料」分開設計。原因是合約內容常有模糊、例外、補充條款或不規則書寫，若直接寫入正式資料表，會增加錯誤風險。應先有一層 `contract_extraction_results` 作為暫存與審核層。

| 欄位分類 | 需抽取欄位 | 備註 |
|---|---|---|
| 客戶主資訊 | 客戶名稱、統編、聯絡人、Email、專案名稱 | 可與既有客戶資料合併 |
| 商務資訊 | 合約總額、是否含稅、稅率、幣別 | 預設先以台幣與 5% 營業稅處理 |
| 付款資訊 | 付款期數、各期比例 / 金額、付款條件、請款條件 | 需支援 3/4/3、6/2/2 與自由文字 |
| 驗收資訊 | 驗收節點名稱、驗收條件、預計日期 | 應轉成 milestones |
| 保固資訊 | 保固天數、起算條件、保固範圍 | 常見為驗收完成後起算 |
| 維護資訊 | 是否有維護費、金額、收費週期、起算條件 | 有些案件無維護費 |
| 內部資訊 | 負責業務、分潤比例、發包需求、特例條款 | 這些可能來自人工補充而非合約 |

### 3.4 付款期程與驗收點的邏輯設計

目前最重要的不是支援某一種固定付款模板，而是支援「從合約語意中抽出付款與驗收規則」。因此建議將付款期程拆成兩層：第一層是**付款排程**，第二層是**付款觸發條件**。這樣即使合約寫法不同，也能以統一資料模型表示。

| 資料實體 | 用途 | 範例 |
|---|---|---|
| `project_payment_schedules` | 儲存每一期付款比例 / 金額 | 第 1 期 30%、第 2 期 40%、第 3 期 30% |
| `project_milestones` | 儲存每個驗收點或交付節點 | 需求確認完成、初版交付、正式上線 |
| `milestone_payment_links` | 建立驗收點與付款期的關聯 | 第 2 期款對應「初版驗收通過」 |
| `project_warranties` | 儲存保固條件與期間 | 驗收後 30 天保固 |
| `project_maintenance_plans` | 儲存維護費起算規則 | 保固結束次日起每月 NT$ X |

### 3.5 分潤計算邏輯：改為依未稅實收金額計算

您目前最明確的業務規則，是分潤必須依**客戶實際付款的未稅金額**計算，而不是依合約總額或含稅金額計算。這一點非常重要，因為它會影響分潤、勞報單與後續帳務全部連動。

建議規則如下。

| 規則項目 | 建議規則 |
|---|---|
| 預設分潤率 | 專案預設 25% |
| 專案客製分潤率 | 每案可覆寫分潤比例 |
| 計算基礎 | 每筆客戶付款的未稅金額 |
| 未稅公式 | `實收含稅金額 ÷ (1 + 稅率)` |
| 本次應計分潤 | `本次未稅實收 × 分潤率` |
| 已預支抵扣 | `本次可發放 = 本次應計分潤 - 已預支待沖銷金額` |
| 多筆付款累計 | 每次收款都獨立產生 commission accrual record |

> 您提供的例子可被系統化為：若某案第一期收款 210,000 元含稅，稅率 5%，未稅金額為 200,000 元；若該案分潤比率為 25%，則本次應計分潤為 50,000 元。這筆資料應自動寫入分潤待發紀錄，並進一步觸發勞報單流程。

### 3.6 勞報單自動化流程

勞報單在您的流程裡不是單純的報表，而是實際發放與簽回的關鍵憑證。因此建議把「分潤發放」與「勞報單產製」做成同一流程中的兩個狀態，而非完全分離。

| 階段 | 系統動作 | 備註 |
|---|---|---|
| 分潤計算完成 | 建立可發放分潤紀錄 | 來源可能是正常分潤或預支 |
| 準備發放 | 計算毛額、扣繳稅、二代健保、實發淨額 | 依規則自動完成 |
| 產生勞報單 | 自動帶入業務個人資料、案件資訊、金額 | 可匯出 PDF |
| 發送通知 | 通知業務下載 / 簽回 | 可站內通知 + Email |
| 回收簽署 | 上傳簽回文件或標記完成 | 建議保留審核狀態 |
| 帳務入帳 | 標記已付款 / 已歸檔 | 可供財務追蹤 |

### 3.7 預支分潤與其他帳務的擴充需求

您提出了一個非常實務的需求：**不是所有付款都是標準分潤，某些情境是中途預支，某些則是額外款項或發包支出。** 這代表現有財務模組若只做「客戶付款 → 分潤 → 勞報單」的單一路徑，將不足以支撐實際營運。

因此建議新增一個更上層的「交易事件」模型，把不同帳務性質統一納管。

| 交易類型 | 是否影響分潤 | 是否需勞報單 | 建議用途 |
|---|---|---|---|
| 客戶收款 | 是 | 否 | 作為計算分潤的來源事件 |
| 正常分潤發放 | 是 | 是 | 正式分潤發放 |
| 分潤預支 | 是 | 是 | 尚未收到足額款前先預支 |
| 額外獎金 / 補貼 | 否或視情況 | 視情況 | 特殊獎勵或補差額 |
| 發包支出 | 否 | 否 | 外包、採購、轉包成本 |
| 維護費收入 | 否 | 否 | 保固後維護收入追蹤 |

## 四、需求主軸 B：川輝 AI 助理升級

### 4.1 目標描述

目前川輝 AI 助理已經具備群組訊息保存、檔案保存與一定程度的身份識別能力，但仍偏向「被動保存」與「事後摘要」。您希望把它升級為真正可操作的工作助理，也就是它不只會看懂，還要會**執行**。

### 4.2 身份辨識與後台人工補正

這是所有後續智能功能的基礎。如果系統不知道群組內的某個人到底是內部員工、VIP 客戶、專案窗口，後面所有的提醒、會議邀請、檔案授權與問答都會失準。因此建議把「身份映射」做成獨立後台能力，而不是散落在綁定流程裡。

| 功能 | 建議設計 |
|---|---|
| 成員主檔 | 建立 `line_identities` 或 `contact_identities` 作為群組身份主表 |
| 人工綁定 | 可手動把 LINE 成員綁到內部員工、客戶、廠商或 VIP |
| 欄位補充 | 姓名、公司、角色、Email、電話、備註、標籤 |
| 信任優先級 | 手動設定覆蓋 AI 自動辨識 |
| 群組脈絡 | 同一 LINE ID 可在不同群組扮演不同身份標記 |

### 4.3 檔案原始檔名保存與下載還原

這是非常明確且應優先處理的基礎問題。您已做到永久保存，但如果下載時檔名與原始不同，實務上會影響找檔、交接與對帳。這個需求的本質不是 UI 問題，而是**儲存 metadata 不完整**。

| 改善項目 | 建議欄位 / 作法 |
|---|---|
| 儲存唯一檔名 | `storage_object_key` |
| 原始檔名 | `original_file_name` |
| MIME 類型 | `mime_type` |
| 副檔名 | `file_extension` |
| 上傳來源訊息 | `source_message_id` |
| 下載輸出 | 後台下載時以 `original_file_name` 作為輸出檔名 |

### 4.4 群組內建立 Google Meet 會議

您希望的不是單純建立一則提醒，而是透過 @川輝AI助理 的自然語言或指令，在群組內完成「解析時間、辨識受邀者、建立會議、回傳連結、必要時寄邀請、會前提醒」的一整串動作。這代表這個功能本質上是一個**指令型工作流**。

| 使用者輸入 | 系統處理 |
|---|---|
| `/建立會議 4/13 (二) 15:00~16:00 所有人` | 解析時間、群組範圍、建立會議、回傳連結 |
| `幫我建立下周三會議 15:00 邀請 @A @B` | 解析相對日期、時段、指定成員 |
| 成員已綁定 Email | 自動加入 Google Calendar 受邀者 |
| 成員無 Email | 仍建立 Meet，並安排群組提醒 |
| 接近會議時間 | 前一天與前一小時回群組提醒並附連結 |

建議此功能拆成四層。

| 層級 | 責任 |
|---|---|
| 指令解析層 | 解析日期、時間、參與者、標題、會議描述 |
| 身份映射層 | 從被 @ 人員或「所有人」找到對應 LINE 身份與 Email |
| 會議建立層 | 建立 Google Calendar Event + Meet 連結 |
| 通知層 | 群組回傳連結、寄出邀請、建立提醒任務 |

### 4.5 群組內取回過期檔案

這個需求非常適合做成標準指令，因為資料已經在你們雲端，只差最後一段「從訊息回應關係找到檔案並重傳」。因此系統關鍵不在 AI，而在於**reply message → source message → stored file** 的映射是否完整。

| 功能步驟 | 系統要求 |
|---|---|
| 使用者回覆歷史訊息 | 需取得原始訊息 ID |
| 助理收到指令 | 辨識此回覆針對哪個附件訊息 |
| 系統查找映射 | 從 line_messages 找到 storage object |
| 權限確認 | 確認操作者具該群組使用權 |
| 檔案重傳 | 從雲端讀取並重新上傳到群組 |

### 4.6 群組知識問答與會議記錄理解

這項需求的價值很高，因為它會把原本只是保存資料的系統，變成真正可詢問的專案記憶體。但要做好，不能只靠單次 LLM 回答，而必須有一個清楚的檢索來源排序。

| 知識來源 | 建議優先級 | 說明 |
|---|---|---|
| 已整理的會議記錄 | 高 | 結構化程度高，可信度較高 |
| AI 自動摘要 | 中高 | 適合抓結論與近期脈絡 |
| 原始群組訊息 | 中 | 可追溯，但雜訊高 |
| 上傳文件 / 合約 / 提案 | 高 | 適合回答正式條件與歷史依據 |
| 專案里程碑 / 任務 | 高 | 適合回答當前狀態與待辦 |

建議問答輸出也分成兩種模式。

| 模式 | 用途 |
|---|---|
| 快速回答模式 | 回答「這題之前有討論過嗎」「目前結論是什麼」 |
| 依據模式 | 回答時附帶出處，例如來自哪次會議、哪份文件、哪段對話 |

### 4.7 群組內直接綁定 Email 與聯絡資訊

這個需求與身份辨識後台其實是同一套模型的兩種入口。後台是管理入口；群組指令是即時操作入口。兩者都應寫入同一份聯絡人主檔，避免資料不一致。

| 輸入方式 | 建議行為 |
|---|---|
| `@川輝AI助理 @客戶A 綁定Email:xxx@gmail.com` | 辨識指定成員與 Email，更新聯絡人資料 |
| 管理員後台編輯 | 可補充姓名、公司、標籤、VIP、內部人員、備註 |
| 驗證邏輯 | Email 格式驗證，必要時可要求確認 |
| 權限控制 | 僅 admin / leader / 指定 PM 可執行 |

## 五、系統缺口分析

從您目前的系統架構來看，這些需求不是做不到，而是目前仍缺少幾個關鍵能力層。

| 缺口 | 目前狀態 | 升級需求 |
|---|---|---|
| 合約理解層 | 有文件解析，但偏向文字抽取與向量化 | 需要結構化商務條款抽取與人工確認流程 |
| 領域模型 | 有 project / commission / labor_receipt，但不足以表達保固、維護、預支、特例 | 需新增合約、付款規則、里程碑、交易事件模型 |
| 財務引擎 | 有基本分潤與勞報單 | 需改為未稅實收驅動，並支援預支與其他款項 |
| 聯絡人主檔 | 有 users 與 LINE 綁定 | 需獨立 contact / identity / participant 模型 |
| 指令執行層 | 有摘要與分析 | 需新增 command parser + workflow executor |
| 檔案 metadata 治理 | 可存檔，但檔名回復不足 | 需補 original filename 與訊息映射 |
| 任務與提醒層 | 有部分 Cron | 需支援會議提醒、維護費起算提醒、驗收提醒 |

## 六、建議新增或調整的資料模型

以下是我建議優先補上的資料結構。這不是最終 ER Model，但足以作為工程規劃起點。

| 資料表 / 模組 | 主要用途 | 關鍵欄位 |
|---|---|---|
| `contracts` | 儲存正式合約主檔 | client_id, project_id, contract_amount, tax_rate, signed_date, file_id |
| `contract_extraction_results` | 儲存 AI 抽取結果與信心分數 | raw_json, normalized_json, status, reviewed_by |
| `project_milestones` | 儲存驗收點 / 交付點 | project_id, title, description, due_date, acceptance_criteria, status |
| `project_payment_schedules` | 儲存付款期程 | project_id, sequence_no, percentage, gross_amount, net_amount, trigger_type |
| `milestone_payment_links` | 驗收點與付款關聯 | milestone_id, payment_schedule_id |
| `project_warranties` | 保固規則 | project_id, warranty_days, start_trigger, end_date |
| `project_maintenance_plans` | 維護費規則 | project_id, enabled, start_rule, monthly_fee |
| `commission_rules` | 每案分潤規則 | project_id, user_id, commission_rate, tax_basis |
| `commission_events` | 分潤來源事件 | project_id, payment_id, gross_received, net_received, calculated_commission |
| `payout_advance_records` | 分潤預支 | project_id, user_id, advance_amount, offset_status |
| `finance_transactions` | 統一記帳事件 | project_id, type, amount, needs_labor_receipt, reference_id |
| `contact_identities` | 身份主檔 | line_user_id, display_name, identity_type, email, is_vip |

## 七、MVP 與完整版切分建議

為了避免一次做太大，我建議先把需求分成 **MVP 階段** 與 **完整版階段**。MVP 專注在立刻能省時間、降低錯誤、且最貼近現在流程的部分；完整版再把智能體驗拉高。

### 7.1 MVP 階段

| 模組 | MVP 功能 |
|---|---|
| 合約建案 | 上傳最終提案書 + 合約後，自動抽取總金額、付款期程、驗收點、保固天數、維護費啟用與否，並進入人工確認建案流程 |
| 專案追蹤 | 自動建立付款期程與驗收里程碑，顯示於案件追蹤頁 |
| 分潤引擎 | 依每筆客戶實收未稅金額計算分潤，支援每案自訂分潤率 |
| 勞報單 | 分潤與預支時自動產生勞報單、稅額與健保費試算、自動帶入業務資料 |
| 帳務補充 | 可建立額外款項 / 發包支出等簡易記帳紀錄 |
| 身份補正 | 後台可手動綁定內部人員、VIP 客戶、Email |
| 檔名還原 | 保存原始檔名，下載時自動還原 |
| 建立會議 | 支援群組指令建立 Google Meet 會議並回傳連結 |
| 會議提醒 | 若找不到 Email，至少支援群組前一天與前一小時提醒 |

### 7.2 完整版階段

| 模組 | 完整版新增功能 |
|---|---|
| 合約理解 | 支援更複雜條款、附約、補充協議、模糊條件推論與多版本比對 |
| 維護費自動化 | 保固期結束後自動建立維護費帳單 / 追蹤維護收入 |
| 財務彈性化 | 支援多人分潤、部分沖銷、補差額、特殊獎金、跨期調整 |
| AI 助理取檔 | 回覆歷史訊息後自動從雲端取回附件並傳回群組 |
| 群組知識問答 | 可回答本群討論結論、待辦、歷史決策，並附依據來源 |
| 群組指令補資料 | 在群組直接綁定 Email、角色、VIP 標記 |
| 管理儀表板 | 顯示 AI 指令成功率、會議建立紀錄、取檔紀錄、辨識信心與人工修正率 |

## 八、優先順序建議

若以「短期最有感」與「最能降低人工成本」為標準，我建議實作順序如下。

| 優先級 | 功能 | 原因 |
|---|---|---|
| P1 | 合約上傳後自動建案 + 人工確認 | 直接解決最耗時的人工整理 |
| P1 | 付款期程 / 驗收點自動建立 | 是後續財務、專案追蹤與提醒的基礎 |
| P1 | 未稅實收分潤 + 勞報單自動產生 | 直接降低財務與人工作業錯誤 |
| P1 | 後台手動綁定身份與 Email | 是 AI 助理準確度的基礎 |
| P2 | Google Meet 指令建立與提醒 | 使用頻率高，對團隊體感明顯 |
| P2 | 檔案原始檔名還原 | 技術複雜度低但體感改善大 |
| P3 | 預支、額外款項、發包記帳統一化 | 會提升財務完整度 |
| P3 | 群組取回過期檔案 | 高價值，但仰賴訊息映射完整 |
| P3 | 群組知識問答 | 高價值，但應在資料結構完善後再做 |

## 九、後台操作設計建議

這批功能若只靠 AI 自動化而沒有好的後台，實務上會很痛苦。建議後台一定要遵循「先自動、再確認、最後允許特例修正」的原則。

| 畫面 | 核心用途 | 介面重點 |
|---|---|---|
| AI 建案確認頁 | 檢查合約抽取結果 | 高亮 AI 不確定欄位、可一鍵修正 |
| 付款與驗收頁 | 檢視付款期程與對應驗收點 | 以時間軸或表格顯示，支援手動插入特例 |
| 分潤中心 | 檢視每筆未稅收款、應計分潤、預支、已發放 | 強調來源依據與沖銷關係 |
| 勞報單中心 | 產單、下載、簽回、歸檔 | 狀態清楚、可批次處理 |
| 身份管理頁 | 綁定 LINE 成員、Email、VIP 標記 | 搜尋快、可從群組脈絡查看 |
| AI 指令紀錄頁 | 查看助理做過哪些事 | 方便排錯與追溯 |

## 十、實作風險與注意事項

| 風險 | 說明 | 建議做法 |
|---|---|---|
| 合約內容格式不一 | AI 抽取容易受文件格式影響 | 建立人工確認層，不直接自動入正式資料 |
| 付款條件語意模糊 | 合約常寫成自由文字 | 先支援常見模板，再支援例外人工修正 |
| 分潤規則例外多 | 有預支、補差額、多人分潤等 | 用事件制模型而非單一計算公式硬寫死 |
| LINE 身份不完整 | 無法準確邀請與提醒 | 先做人工補綁與權限角色管理 |
| 取回檔案需訊息映射 | 若早期資料未保存完整關聯，功能會受限 | 新資料先完整保存，舊資料可逐步補齊 |
| 問答可信度 | 若沒有出處，容易產生誤判 | 回答應附會議、文件或訊息來源 |

## 十一、結論

整體來看，您這次提出的需求不是零散的小功能，而是一個很清楚的升級方向：把目前的系統從「記錄型內部平台」升級成「可執行的營運中台 + 群組智能助理」。

其中第一塊的核心，在於讓**合約變成案件、付款變成分潤、分潤變成勞報單、保固結束變成維護計畫**。第二塊的核心，在於讓**川輝 AI 助理從保存資料，進化成能辨識身份、建立會議、取回檔案、回答問題的群組工作代理人**。

如果只問我一句最務實的建議，我會建議您下一步優先做三件事：**第一，先完成合約抽取後的人機協作建案流程；第二，先把未稅分潤與勞報單流程重構完成；第三，先把 LINE 成員身份主檔與手動補綁能力建立起來。** 這三件事完成後，後面的會議建立、取檔、問答，才會真正穩。
