## 二、系統現況總結

依據您原先提供的架構文件，川輝科技目前的內部系統已經不是單一 CRM 或單一專案系統，而是一個結合 **商機管理、專案管理、付款追蹤、分潤計算、勞報單產出、LINE 群組訊息留存、會議記錄分析，以及 AI 文件生成** 的營運型平台。這代表它最大的價值，不只是把資料數位化，而是已經開始把公司核心營運流程收斂到同一個系統中。

| 面向 | 現況判讀 | 意義 |
|---|---|---|
| 系統定位 | 營運中台型內部系統 | 承接業務、專案、財務、知識管理與協作 |
| 架構型態 | Next.js + Supabase + Vercel 的輕量 SaaS 組合式架構 | 適合快速迭代，但需補治理 |
| 既有 AI 能力 | 文件解析、RAG、會議分析、對話摘要 | 已具備升級成 AI Agent 的基礎 |
| 既有 LINE 能力 | 訊息永久保存、檔案保存、群組管理 | 已具備延伸成工作助理的起點 |
| 既有財務能力 | 付款、分潤、勞報單已存在 | 可延伸為更完整的自動化財務引擎 |

## 三、目前架構優勢與限制

這套系統的優勢，在於功能已高度貼近實際業務流程，也已具備資料沉澱、AI 輔助與流程串接能力；但限制也非常明顯：越是進入財務、合約、AI 操作與外部協作層，越需要更完整的**規則集中化、資料模型治理、可觀測性與權限控制**。

| 類別 | 優勢 | 限制 |
|---|---|---|
| 業務流程 | 從商機到分潤形成閉環 | 某些步驟仍需人工轉譯文件內容 |
| 財務流程 | 已有分潤與勞報單基礎 | 規則仍偏人工、特例處理不足 |
| AI 能力 | 已有 RAG 與生成能力 | 尚未升級成可直接操作系統的 Agent |
| LINE 助理 | 已能保存資料與辨識部分身份 | 尚未具備完整執行指令與高精度身份治理 |
| 技術架構 | 上線快、維護成本相對低 | 權限、資料層規則、後台治理仍需強化 |

## 四、整體升級方向

本次整體升級可以明確分成三大主軸，而且這三塊其實彼此相連，不應視為獨立功能。

| 主軸 | 核心問題 | 升級目標 |
|---|---|---|
| 主軸 A：合約與帳務自動化 | 合約內容需人工整理、付款與分潤需人工計算 | 讓合約直接變成案件、付款計畫、驗收點、分潤與勞報單 |
| 主軸 B：川輝AI助理強化 | LINE 助理知道資料但不夠能做事 | 讓助理能辨識、建立會議、取回檔案、回答問題、補綁資訊 |
| 主軸 C：AI Chat / AI Agent 操作層 | 使用者仍需自己進系統逐步操作 | 讓使用者能直接用自然語言要求系統完成工作 |

## 五、主軸 A：合約、專案、帳務與分潤自動化

### 5.1 核心目標

您希望在上傳**最終提案書 + 合約**之後，系統就能自動判讀並建立該案，而不是再由內部人員手動補每一個欄位。這包含專案基本資料、總金額、付款期程、驗收點、整體期程、保固條件、維護費起算、分潤規則、勞報單與其他記帳事件。

### 5.2 自動建案應抽取的內容

| 類別 | 應自動整理的資訊 |
|---|---|
| 專案主資料 | 客戶名稱、專案名稱、簽約日期、總金額、稅別、專案期程 |
| 付款條件 | 付款期數、付款比例、每期條件、請款條件 |
| 驗收條件 | 各階段驗收點、驗收標準、對應時間或付款節點 |
| 保固與維護 | 保固天數、起算基準、是否有維護費、維護費金額與起算時點 |
| 業務與分潤 | 負責業務、分潤比率、特例規則 |
| 附件與會議 | 關聯提案書、合約、會議記錄、補充文件 |

### 5.3 付款期程與驗收點

您提到客戶付款方式有時是 3/4/3，有時是 6/2/2，也可能依合約寫法不同，因此不能採固定模板。系統需要把合約中的商務條件轉成標準化的付款期程與驗收節點。

| 需求 | 期望結果 |
|---|---|
| 自動辨識付款比例 | 能識別 30/40/30、60/20/20 或其他比例 |
| 自動建立付款期程 | 每一期需有比例、金額、條件、預計日期 |
| 自動建立驗收點 | 驗收條件與專案里程碑同步建立 |
| 驗收與付款綁定 | 某期款需能對應驗收通過、上線、交付等事件 |

### 5.4 保固與維護費邏輯

系統需支援不同案件有不同保固期，也要支援某些案件沒有維護費。保固與維護費不應只是備註，而應變成系統可運算的條件。

| 情境 | 系統規則 |
|---|---|
| 有保固、有維護費 | 先跑保固期，保固期滿後自動啟用維護計畫 |
| 有保固、無維護費 | 只追蹤保固到期，不建立維護收費 |
| 無保固、有維護費 | 依約定日期或結案日直接開始維護計畫 |
| 無保固、無維護費 | 僅保存合約條件供查詢 |

### 5.5 分潤與勞報單自動化

您已明確定義分潤規則：**依客戶實際付款的未稅金額計算分潤**，每案分潤比例可能不同，預設可為 25%。這個規則必須成為正式引擎邏輯。

> 例如：若第一期收款 210,000 元含稅，稅率 5%，未稅金額為 200,000 元；若該案分潤比率 25%，本期應計分潤為 50,000 元。系統應自動建立分潤事件、產出勞報單、計算扣繳與二代健保，並進入通知與簽回流程。

| 項目 | 系統規則 |
|---|---|
| 分潤比例 | 每案可設定，預設 25% |
| 分潤基礎 | 依每筆實收未稅金額計算 |
| 勞報單產出 | 分潤或預支發生時自動產生 |
| 稅務計算 | 自動扣 10% 所得稅與 2.11% 健保費 |
| 送件流程 | 自動帶入業務資料、通知下載、簽回、歸檔 |

### 5.6 特例與簡易記帳

除了正常分潤外，您也提到**預支分潤、額外款項、發包款項**等需求，這表示財務模型必須從單一路徑改成「事件型交易模型」。

| 交易類型 | 是否需勞報單 | 需求 |
|---|---|---|
| 正常分潤 | 是 | 依收款自動計算與產單 |
| 分潤預支 | 是 | 可提前發放並後續沖銷 |
| 額外款項 | 視情況 | 可簡單記帳與歸類 |
| 發包款項 | 否 | 記錄支出與對應案件 |
| 維護費收入 | 否 | 追蹤保固後持續性收入 |

## 六、主軸 B：川輝AI助理升級

### 6.1 身份辨識與後台手動補正

您希望若內部人員沒有綁定、或某些 VIP 客戶需要被特別辨識，管理者可以直接在後台手動設定。這是合理且必要的，因為很多 AI 能力都建立在身份辨識正確的前提上。

| 功能 | 說明 |
|---|---|
| 手動綁定身份 | 可將 LINE 成員綁定為內部人員、客戶、VIP、窗口等 |
| 補充聯絡資料 | 可輸入 Email、公司、姓名、角色、備註、標籤 |
| 優先級控制 | 手動設定結果應覆蓋 AI 自動推論 |
| 搜尋與管理 | 可由群組、姓名、LINE 顯示名稱快速管理 |

### 6.2 檔案原始檔名還原

目前系統雖然已永久保存群組檔案，但下載時檔名不一致，這會影響管理與找檔。建議未來儲存「內部唯一鍵」與「原始檔名」分離。

| 欄位 | 用途 |
|---|---|
| `storage_key` | 雲端儲存唯一鍵 |
| `original_file_name` | 原始上傳檔名 |
| `source_message_id` | 對應原始 LINE 訊息 |
| `mime_type` | 檔案類型 |
| `uploaded_by_identity_id` | 上傳者身份 |

### 6.3 群組內建立 Google Meet 會議

您希望能直接在 LINE 群組裡用自然語言或指令請川輝AI助理建立會議，例如：

> `@川輝AI助理 /建立會議 4/13 (二) 15:00~16:00 所有人`

或：

> `@川輝AI助理 幫我建立下周三會議 15:00 邀請 @A @B`

系統應能解析時間、辨識成員、建立 Google Meet、回傳連結，若已有 Email 便同步寄出邀請；若沒有 Email，也至少能在前一天與前一小時於群組中提醒。

| 能力 | 說明 |
|---|---|
| 時間解析 | 支援明確日期與相對日期 |
| 參與者辨識 | 支援所有人、@指定人員、已綁定對象 |
| 建立會議 | 建 Google Calendar Event + Meet 連結 |
| 自動邀請 | 有 Email 時自動邀請 |
| 群組提醒 | 無 Email 時以群組提醒補足 |

### 6.4 群組內取回過期檔案

當群組中的 LINE 原始檔案連結已過期，但系統後端仍保留該檔案時，使用者應能直接回覆那則訊息並 @川輝AI助理，要求系統自動從雲端取回並重傳到群組。

| 步驟 | 系統要求 |
|---|---|
| 回覆歷史訊息 | 取得原始訊息 ID |
| 辨識對應附件 | 找到對應雲端儲存檔案 |
| 權限檢查 | 確認使用者屬於該群組且有取檔權限 |
| 重傳檔案 | 將檔案重新送回群組 |

### 6.5 群組知識問答與會議記錄理解

您希望川輝AI助理能回答本群組過去討論過的事情，包括某個問題之前的結論、待辦事項、負責人、會議內容等。這部分應建立在群組摘要、會議記錄、文件知識庫與歷史訊息的整合檢索之上。

| 問題類型 | 助理應能回答 |
|---|---|
| 本群之前討論過什麼 | 整理近期與歷史關鍵結論 |
| 某專案目前卡在哪 | 結合群組討論、會議記錄與案件狀態回答 |
| 某次會議決議是什麼 | 回答摘要、待辦與負責人 |
| 某份文件在哪裡 | 指出文件來源與關聯訊息 |

### 6.6 群組內直接補綁 Email

您提到可以在群組中直接下指令：

> `@川輝AI助理 @客戶A 綁定Email:anb@gmail.com`

這類指令非常實用，因為能把管理工作從後台延伸到群組現場，提高資料即時性。

| 功能 | 規則 |
|---|---|
| 指令辨識 | 能辨識被標註對象與 Email |
| 後台同步 | 直接更新聯絡人主檔 |
| 權限控制 | 僅管理者 / PM / 指定角色可執行 |
| 驗證機制 | Email 格式驗證，必要時要求確認 |

## 七、主軸 C：AI Chat / AI Agent 操作內部系統

### 7.1 核心目標

您希望公司內部系統本身也能有 **AI Chat**，也就是使用者不必每次自己點選頁面、填欄位、找資料，而是可以直接用自然語言告訴系統想做什麼，由 **AI Agent** 幫忙完成操作。這將會是整個系統最有感的升級之一。

### 7.2 AI Chat 的定位

這個 AI Chat 不應只是「對話視窗」，而應是**具備系統操作能力的工作代理介面**。也就是說，使用者可以說：

> 幫我建立一個新案件，客戶是某某公司，總金額 35 萬，分三期收款。

或：

> 幫我找出這個月還沒請款但已達驗收點的案件。

或：

> 幫我把這次會議內容整理成提案規格草稿。

系統應該能理解意圖、補問缺漏、顯示待確認結果，最後再執行。

| AI Chat 能力類型 | 作用 |
|---|---|
| 查詢型 | 查資料、查狀態、查歷史決策、查待辦 |
| 生成型 | 生成提案書、規格書、會議摘要、報價草稿 |
| 操作型 | 建案、建會議、補資料、更新狀態、建立任務 |
| 分析型 | 彙整案件風險、預估分潤、找出卡關專案 |

### 7.3 AI Agent 可操作的範圍

建議一開始不要讓 AI Agent 無限制操作全部功能，而是先定義清楚「可自動執行」、「需確認後執行」與「僅可建議不可執行」三種層級。

| 操作層級 | 範例 | 安全建議 |
|---|---|---|
| 直接執行 | 查詢案件、整理摘要、生成草稿 | 低風險，可直接回應 |
| 確認後執行 | 建案、建立付款期程、建立會議、更新 Email、產生提案書 | 執行前需顯示確認畫面 |
| 嚴格限制 | 發放分潤、刪除資料、修改財務數字、對外發送正式文件 | 需權限 + 二次確認 + 留存稽核 |

### 7.4 權限與安全邊界

您也明確提到：川輝AI助理之後可以擁有這種能力，但一定要有安全的權限判定。這點非常重要。AI Agent 若沒有權限邊界，就會變成風險來源，而不是效率工具。

| 安全機制 | 建議作法 |
|---|---|
| 角色權限 | 依 admin / finance / leader / pm / sales 設定可操作範圍 |
| 動作分級 | 查詢、編輯、財務、外發、刪除分級管理 |
| 二次確認 | 重要動作需顯示「即將執行內容」由使用者確認 |
| 稽核紀錄 | AI 每一次查詢、產生、修改、發送都要留記錄 |
| 可回溯性 | 回答與操作都要有來源依據與操作者身份 |
| 範圍限制 | 僅能存取使用者本身有權限看到的資料 |

### 7.5 建議的 AI Agent 執行流程

| 階段 | AI Agent 行為 |
|---|---|
| 理解需求 | 解析自然語言意圖 |
| 查找資料 | 從系統資料、知識庫、群組摘要中找上下文 |
| 補問缺漏 | 若資訊不足，主動追問必要欄位 |
| 形成計畫 | 轉成一組系統可執行的步驟 |
| 顯示確認 | 若涉及建立或修改，先顯示預計動作 |
| 執行操作 | 寫入資料、建立文件、發起流程 |
| 回報結果 | 告知完成狀態、產物與後續步驟 |

## 八、川輝AI助理也應具備同樣的 Agent 能力

您特別提到，這個能力未來不只在內部系統 Chat 裡有，**川輝AI助理也要能擁有同樣的能力**。這代表系統應該採取「同一套 Agent 核心，不同入口介面」的設計。

| 入口 | 使用情境 | 背後能力應否共用 |
|---|---|---|
| 內部系統 AI Chat | 員工在後台直接對話操作 | 是 |
| 川輝AI助理（LINE OA） | 員工在群組或私訊中直接要求執行 | 是 |
| 管理後台 | 人工校正與監控 Agent 行為 | 是 |

也就是說，未來應該建立統一的：

| 核心層 | 功能 |
|---|---|
| Intent / Command Parser | 理解自然語言意圖 |
| Permission Engine | 判斷是否可執行 |
| Workflow Engine | 執行建案、會議、文件生成等流程 |
| Knowledge Retrieval Layer | 從提案書、會議、文件、對話中取知識 |
| Audit Log | 留存所有 Agent 行為紀錄 |

## 九、業務可透過川輝AI助理生成提案書與討論規格

這是非常關鍵的一塊，也是未來最能放大效率的功能之一。您希望業務能直接透過川輝AI助理產生提案書、討論規格，而不需要每個人再各自去對接不同工具。這代表川輝AI助理要從「記錄型助手」升級成「知識型售前助理」。

### 9.1 業務工作場景

| 場景 | 業務可直接對川輝AI助理說 |
|---|---|
| 產提案書 | 幫我根據某客戶需求產一版提案書 |
| 討論規格 | 幫我整理這次會議內容，列出 MVP 規格 |
| 比對歷史案例 | 找三個最像的歷史提案給我參考 |
| 補強說法 | 幫我把這一段價值主張寫得更完整 |
| 形成報價前草稿 | 根據需求先列功能範圍與建議階段 |

### 9.2 知識來源

您特別指出，希望業務可以享有您已經長期累積的知識庫，包括 Manus 的使用脈絡，以及公司現在所有發出去的提案書。從系統規劃角度，這表示要建立一個**公司級售前知識層**。

| 會議記錄 | 作為客戶需求與上下文來源 |
| 群組對話 | 作為需求補充與決策歷程來源 |
| 您過去累積的 Manus 工作成果 | 作為高品質提案寫法、分析邏輯與文件模板來源 |

### 9.3 提案生成能力定位

未來的提案生成，不應只是一鍵產生長文，而應是具備「找案例、抓需求、產草稿、可協作修改」的完整工作流。

| 能力 | 說明 |
|---|---|
| 相似案例檢索 | 找出最像的既有提案書與規格 |
| 需求重組 | 把會議記錄與自然語言需求整理成結構化需求 |
| 提案草稿生成 | 依公司標準語氣與結構產生第一版 |

| 文件版本管理 | 每次修改保留版本，避免覆蓋 |
| 權限控管 | 僅能取用有權限存取的案例與文件 |

## 十四、後台與操作體驗建議

未來後台不能只是資料表單集合，而應是「讓 AI 幫你做大部分、人工只補必要例外」的工作台。尤其這種系統一旦功能擴張，若後台不好用，大家最後還是會退回人工 Excel 或 LINE 私聊。

| 畫面 | 目標 | 體驗重點 |
|---|---|---|
| AI 建案確認頁 | 檢查合約抽取結果 | 高亮不確定欄位、快速修正 |
| 分潤與帳務中心 | 管理應發、預支、沖銷、勞報單 | 金額來源清楚、流程狀態清楚 |
| 身份管理頁 | 綁定 LINE / Email / VIP / 角色 | 搜尋快、群組脈絡清楚 |
| AI Chat 工作台 | 直接下自然語言命令給系統 | 可看理解結果、可看待確認動作 |
| 助理執行紀錄頁 | 查 AI 做了什麼 | 方便追責、排錯、稽核 |
| 業務提案工作台 | 生成提案、拉歷史案例、討論規格 | 支援版本與知識引用 |

## 十五、關鍵風險與治理建議

| 風險 | 說明 | 建議 |
|---|---|---|
| AI 誤判合約內容 | 付款、驗收、保固條件若抽錯會影響後續全部流程 | 必須有人工確認層 |
| AI Agent 權限過大 | 可能誤改財務或客戶資料 | 需嚴格角色權限與二次確認 |
| 歷史知識庫品質不一 | 提案生成可能引用過時內容 | 建立版本、狀態與可見範圍管理 |
| 群組身份辨識錯誤 | 會影響通知、問答與邀請 | 建立人工補正與覆蓋機制 |
| 取回檔案與發送動作 | 涉及資料外發與權限 | 必須做操作審計與群組權限檢查 |
| 售前知識共用 | 業務能否看到所有案例需有限制 | 設計角色範圍與文件等級 |

## 十六、結論

整體來看，川輝科技這套內部系統已經具備非常好的基礎，不是從零開始，而是已經走到一個關鍵升級點。現在最重要的，不只是再多加幾個功能，而是把系統升級成三個角色同時成立的平台：

第一，它要是**營運中台**，能把合約、付款、分潤、勞報單與維護計畫串起來。第二，它要是**群組工作助理**，讓川輝AI助理真正能在 LINE 中辨識、提醒、建會議、找檔與回答問題。第三，它要是**知識化 AI Agent 平台**，讓內部使用者與業務都能直接用自然語言要求系統做事、生成提案、討論規格，並共享公司累積的提案與文件知識資產。

如果要我給最務實的下一步建議，我會建議您先做四件事：**一，建立合約抽取後的人機協作建案流程；二，重構未稅分潤與勞報單引擎；三，建立聯絡人 / 身份主檔與權限模型；四，先做一版可查詢、可生成、可確認後執行的 AI Chat MVP。** 這四步一旦完成，後面的川輝AI助理進階能力、業務提案知識化與全系統 Agent 化，就會非常順。
